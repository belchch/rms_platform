# Руководство по тестированию Go-бэкенда (с ИИ)

В этом документе зафиксирована философия тестирования в проекте RMS Platform. Она объединяет строгие академические подходы (в духе «Effective Software Testing» Маурисио Аниче) с прагматикой экосистемы Go и современными возможностями ИИ (Cursor).

---

## 1. Теоретический срез: Классика инженерии в реалиях Go

Подходы к тестированию в Go имеют свой характерный стиль. Вместо тяжелых фреймворков и сложных графов потока управления, язык делает ставку на простоту, плоский код и стандартные инструменты.

| Принцип из классики | Как это реализуется в Go | Роль ИИ |
| :--- | :--- | :--- |
| **Specification-based** (Классы эквивалентности) | **Табличные тесты (Table-driven tests).** Массив структур с входными параметрами и ожидаемым результатом. | Идеально генерирует шаблоны таблиц и заполняет базовые классы эквивалентности (нули, границы, длинные строки). |
| **Structural Testing** (Покрытие веток, MC/DC) | **Statement Coverage + Ранний возврат.** В Go встроен только построчный счетчик. Вместо сложных условий (`A && (B || C)`) пишут цепочки простых `if`. | Указывает непокрытые строки и подбирает входные данные в таблицу, чтобы зайти в недостающую ветку (например, ветку обработки ошибки). |
| **Test Doubles** (Моки, шпионы) | **Fakes/Stubs через маленькие интерфейсы.** Вместо Mockito используются интерфейсы на 1-2 метода, реализуемые ручными структурами-заглушками. | Мгновенно пишет ручные Fake-структуры по интерфейсу, сохраняющие историю вызовов или возвращающие заданные ошибки. |
| **Property-based** (Генерация случайных данных) | **Fuzzing (`go test -fuzz`).** Встроено в язык, само генерирует мусорные и пограничные данные для поиска паник. | ИИ пишет каркас Fuzz-теста, но **вы** должны продумать инварианты (какие свойства не должны меняться при любых данных). |

## 2. Мутационное тестирование (Mutation Testing)

Классическое покрытие (Coverage) отвечает на вопрос: *«какие строки кода выполнились при тестировании?»*
Мутационное тестирование отвечает на вопрос: *«насколько хороши наши тесты? Заметят ли они ошибку?»*

### Как это работает:
1. Инструмент берет ваш код и намеренно портит его (создает «мутантов»): меняет `==` на `!=`, `+` на `-`, удаляет вызовы функций.
2. Запускаются ваши тесты.
3. Если тесты прошли успешно (зеленые) — значит, тест плохой, мутант «выжил» (тест не заметил ошибку бизнес-логики).
4. Если тесты упали — мутант «убит» (тест хороший).

### Мутационное тестирование в Go:
В Go инструменты мутационного тестирования (например, `gremlins` или `go-mutesting`) используются крайне редко, так как они замедляют сборку в сотни раз (код нужно перекомпилировать для каждого мутанта). 

**Как мы это компенсируем в RMS:**
Вместо автоматических мутаций мы используем ИИ. ИИ склонен писать assert-ы, которые ничего не проверяют (утверждают то, что и так мокается). **Ответственность разработчика — проверять, что тест упадет, если в бизнес-логике изменить критический оператор.**

## 3. Практический чек-лист: Тесты в связке с ИИ

Ниже приведен алгоритм написания тестов от простого к сложному с использованием ИИ. Главное правило: **ИИ пишет рутину, вы контролируете архитектуру и смысл.**

### Шаг 1. Выбор цели (изоляция)
Начинайте с чистой логики, не зависящей от базы данных.
*   **Хорошая цель для старта:** Валидаторы, парсеры, утилиты, сложная бизнес-математика, мапперы DTO.
*   **Плохая цель для старта (Unit):** SQL-запросы, Huma-контроллеры, которые делают только CRUD (для них лучше сразу писать интеграционные тесты с реальной БД).

### Шаг 2. Подготовка промпта (Контекст для ИИ)
Откройте файл в Cursor и выделите нужную функцию.
*   **Плохой промпт:** «Напиши тесты для этого файла».
*   **Хороший промпт:** «Напиши табличный тест для функции `CalculatePlanArea`. Учти бизнес-требования: пустой план дает 0, план с пересекающимися стенами возвращает `ErrInvalidGeometry`. Используй `testify/require` для проверок.»

### Шаг 3. Генерация каркаса и заглушек
ИИ сгенерирует слайс `tests := []struct{...}` и цикл `t.Run`. 
*   Если функция зависит от интерфейса (например, `ProjectRepository`), попросите: *«Сгенерируй ручной Fake для `ProjectRepository`, который позволяет задать возвращаемую ошибку для каждого тест-кейса»*.
*   Не соглашайтесь на использование тяжелых библиотек мокирования (типа `gomock`), если хватает простой структуры.

### Шаг 4. Аудит бизнес-смысла (Ваш выход)
Внимательно просмотрите сгенерированные кейсы.
1.  **Достаточно ли кейсов?** ИИ часто забывает краевые случаи. Попросите: *«Добавь тест на ситуацию, когда `workspaceId` из JWT не совпадает с `workspaceId` проекта»*.
2.  **Адекватны ли проверки?** ИИ может написать `require.NotNil(t, result)` вместо того, чтобы проверить реальные поля результата. Заставьте его проверять суть.

### Шаг 5. Запуск и замер покрытия
Запустите команду из корня проекта:
```bash
just check-coverage
```
Посмотрите на цифры. Если функция покрыта на 100% — отлично. Если нет, переходите к шагу 6.

### Шаг 6. Добивание до 100% (Работа с профилем)
Если инструмент показал, что строка 45 не покрыта (например, обработка специфичной ошибки).
1.  Выделите эту непокрытую ветку в редакторе.
2.  Запрос к ИИ: *«Ветка `if err == ErrConflict` не покрыта тестами. Какой кейс добавить в нашу таблицу, чтобы триггернуть эту ошибку?»*
3.  Вставьте предложенный кейс, запустите тесты снова. Цель: 100% Statement Coverage для тестируемой функции.

### Шаг 7. Интеграционное тестирование (Для БД и API)
Если логика функции — это 90% перекладывание данных из базы в HTTP ответ (huma-handlers), табличные unit-тесты дадут мало пользы.
1.  Переключитесь на интеграционное тестирование (как в `apps/api/internal/handler/auth/handler_integration_test.go`).
2.  Промпт к ИИ: *«Напиши интеграционный тест для хендлера. Тест должен использовать реальное соединение с БД. Добавь билд-тег `//go:build integration` в начале файла.»*
3.  Запуск: `just check-integration`.
