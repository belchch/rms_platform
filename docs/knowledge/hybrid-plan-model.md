# ВАЖНО К ПРОЧТЕНИЮ ДЛЯ ИИ (AI INSTRUCTION)
При работе с сущностями чертежа (Plan, Room, Wall) и их синхронизацией всегда учитывай гибридную модель хранения. Не пытайся нормализовать геометрию в реляционные таблицы и не пытайся искать метаданные комнат внутри JSON-геометрии.

## Суть гибридной модели

Полный набор данных чертежа собирается из трёх независимых частей, которые склеиваются на клиенте:

1. **`Plan.payloadJson` (Геометрия)**
   - **Что хранит:** Координаты углов, длины стен, проёмы (двери/окна), топологические связки `wall` ↔ `corner`.
   - **Почему так:** Гарантирует атомарность графа. Геометрия не развалится при частичном сбое сети. LWW-синхронизация применяется ко всему документу целиком. Метаданные (имена комнат, материалы стен) сюда **не сериализуются**.

2. **Таблица `rooms` (Метаданные комнаты)**
   - **Что хранит:** `id`, `plan_id`, `name` (например, «Кухня»), `sync_cursor`, `deleted_at`.
   - **Почему так:** Даёт стабильный ID для привязки фотографий (через `photoables`) и позволяет переименовывать комнату без перезаписи всей геометрии чертежа.

3. **Таблица `walls` (Метаданные стены)**
   - **Что хранит:** `id`, `room_id`, `sync_cursor`, `deleted_at`. В будущем: `material`, `thickness`.
   - **Почему так:** Даёт стабильный ID, чтобы пользователь мог прикрепить фото дефекта к конкретной стене.

## Правило синхронизации
При синхронизации (`/sync/push` и `/sync/pull`) `plan`, `room` и `wall` выступают как **отдельные независимые агрегаты**. Мобильный клиент отправляет их отдельными операциями в батче.

## Склейка (только на клиенте)
В мобильном приложении `PlansRepository.watchById` делает `combineLatest` из локальных таблиц планов, комнат и стен. Имена из таблицы `rooms` подмешиваются в геометрию из `payloadJson` по совпадению `id` перед передачей в редактор Flame.
