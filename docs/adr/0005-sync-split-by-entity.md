# ADR 0005: Разбиение `handler/sync/` по сущностям

## Статус

Принято — 2026-05-14. Заменяет шаги 1 и 2 раздела «Применение» ADR-0004 в части пакета `sync/`. Не отменяет правил ADR-0004 для `auth/` и `photos/`.

## Контекст

ADR-0004 ввёл конвенцию расслоения handler-пакета на transport / orchestration / pure domain через **именование файлов в одном пакете**. Конвенция применима к маленьким и средним пакетам, но не учитывает рост по второй оси — **количеству файлов**.

Фактическое состояние `internal/handler/sync/`:

- 26 файлов: 5 `push_<entity>.go` + 6 `pull_<entity>.go` (с `pull_utils.go`) + 5 `push_<entity>_test.go` + `handler.go`, `push.go`, `pull.go`, `service.go`, `domain.go`, `time.go` + 3 файла общих тестов.
- После backfill ADR-0003 (pull-тесты, service_test, domain_test, handler_test) — **34+ файла**.
- Cohesion слоёв низкая: при работе над `plan` читаются 6 файлов `plan`-логики, остальные 28 — шум. При работе над `lwwWins` читается `domain.go`, остальные 33 — шум.

Конвенция «по слоям внутри одного пакета» в Go-сообществе считается неидиоматичной: Go группирует код **по концепции** (что), не **по слою** (как). При размере пакета свыше ~15 файлов вертикальная нарезка по слоям проигрывает горизонтальной по сущности на двух осях: навигация (где лежит логика `plan`) и независимая работа (конфликт в `push.go`/`pull.go` на любое изменение).

Триггер пересмотра, заложенный в ADR-0004 «Открытые вопросы → 1», сформулирован по размеру одного файла (`service.go` / `domain.go` становится трудно читаемым). Эта формулировка не покрывает рост пакета по количеству файлов и пропускает текущий случай `sync/`.

## Решение

Пакет `internal/handler/sync/` разбивается на корневой transport-пакет и подпакеты по сущности. Состоит из четырёх правил.

### Правило 1. Граница «корень / подпакет» проходит по транзакции

Корневой пакет `sync/` владеет HTTP-эндпойнтом и транзакцией. Подпакет одной сущности владеет логикой push/pull этой сущности и принимает `db.Querier`, не открывает транзакций.

| Файл | Слой | Зависимости |
|---|---|---|
| `sync/handler.go` | transport | `huma.API`, `*pgxpool.Pool`, типы `*Input`/`*Output`, `Register` |
| `sync/push.go` | transport | открытие транзакции, savepoint, диспатч `applyPushOperation` |
| `sync/pull.go` | transport | открытие read-only транзакции, параллельные `List*Since`, сборка `PullChange[]` через `<entity>.AppendPullChanges` |
| `sync/syncdomain/` | pure | LWW, валидация workspace, кросс-сущностные resolver'ы, типы шагов, маппинг времени |
| `sync/<entity>/` | orchestration + per-entity pure | push (Create/Update/Delete), pull-snapshot и append, валидация payload |

Транспорт диспатчит push через `switch op.EntityType` и зовёт `project.ApplyPush(ctx, q, wsID, op)` (или аналог для других сущностей). Pull собирает changes через `<entity>.AppendPullChanges(changes, rows)`.

### Правило 2. Подпакет одной сущности — единый файл, если ≤ ~400 строк

Подпакет `sync/<entity>/` содержит один файл `<entity>.go` со всей логикой сущности: push (upsert + delete), snapshot, append к pull. Один тестовый файл `<entity>_test.go` с обоими сценариями (push + pull).

Раскладка по нескольким файлам внутри подпакета — только при превышении ~400 строк или при появлении логически независимых групп функций. До порога — единый файл.

### Правило 3. `syncdomain/` содержит только кросс-сущностное pure

В `syncdomain/` живёт логика, общая для двух и более сущностей:

- LWW-сравнение (`LWWWins`).
- Валидация workspace (`ValidateWorkspace`).
- Маппинг `int64` ↔ `pgtype.Timestamptz`.
- Резолверы `WorkspaceOfPlan` / `WorkspaceOfRoom` / `WorkspaceOfWall` — кросс-сущностные по определению.
- Типы `PushStepResult` и helper `InternalPushErr(op, err) PushStepResult` — внутренний контракт между подпакетом сущности и transport-слоем.

`syncdomain` не импортирует `huma`, не открывает транзакций, принимает `db.Querier` параметром у функций, которым нужен доступ к БД.

Snapshot-функция одной сущности (`planSnapshot`, `projectSnapshot`) живёт в подпакете сущности, не в `syncdomain` — она привязана к концепции одной таблицы.

### Правило 4. Подпакеты сущностей не зависят друг от друга

`sync/plan/` не импортирует `sync/project/` и наоборот. Резолверы родителя (`WorkspaceOfPlan` зовёт `q.GetProjectByID`) живут в `syncdomain` и принимают `db.Querier`. Зависимость подпакета сущности — только на `syncdomain`, `db`, `synctypes`.

При появлении кросс-сущностной логики (например, каскадного удаления `plan → rooms → walls`) — она оформляется как функция в `syncdomain`, не как импорт между подпакетами.

## Применимость к существующим пакетам

| Пакет | Соответствие сейчас | Что нужно сделать |
|---|---|---|
| `sync/` | нарушено: 26 файлов в плоской раскладке, рост ожидается | разбиение по этому ADR |
| `auth/` | соответствует ADR-0004 после раскладки на `service.go`/`domain.go`; разбиение по сущности неприменимо (одна сущность — `refresh_token`) | без изменений, выполнить раскладку по ADR-0004 |
| `photos/` | соответствует ADR-0004 Правилу 4 (порог не достигнут) | без изменений, выполнить переименование `Handler` → `handler` по ADR-0004 |

Новый ADR заменяет ADR-0004 в части `sync/`. Раздел «Применение» ADR-0004 для `sync/` (шаги 1 и 2) более не действует. Шаги 3 (`auth/`) и 4 (`photos/`) ADR-0004 действуют без изменений.

## Применение

Переход выполняется семью независимыми шагами. Каждый шаг самостоятелен, проверяется через `just check` и мёрджится отдельно. Порядок фиксирован: подпакеты сущностей зависят от `syncdomain`, transport зависит от подпакетов.

1. **`sync/syncdomain/` — выделение pure-ядра.** Создаётся подпакет с экспортируемыми `LWWWins`, `ValidateWorkspace`, `EpochMsToTimestamptz`, `TimestamptzEpochMs`, `TimestamptzEpochMsPtr`, `WorkspaceOfPlan`/`Room`/`Wall`, `PushStepResult`, `InternalPushErr`. Соответствующие unexported имена в корневом `sync/` удаляются, вызовы заменяются. Поведение и сигнатуры — без изменений.
2. **`sync/project/` — миграция первой сущности.** Создаётся `sync/project/project.go` с публичными `ApplyPush(ctx, q, wsID, op) syncdomain.PushStepResult` и `AppendPullChanges(changes, rows) ([]PullChange, error)`. Удаляются `sync/push_project.go`, `sync/pull_project.go`. Тесты переносятся в `sync/project/project_test.go`. `applyPushOperation` в `sync/service.go` зовёт `project.ApplyPush`. `pull.go` зовёт `project.AppendPullChanges`.
3. **`sync/plan/` — миграция.** Аналогично шагу 2.
4. **`sync/room/` — миграция.** Аналогично шагу 2.
5. **`sync/wall/` — миграция.** Аналогично шагу 2.
6. **`sync/photo/` — миграция.** Аналогично шагу 2.
7. **Корневой `sync/` — финальная очистка.** Из корневого пакета удаляются `service.go` (диспатч переезжает в `push.go`), `domain.go`, `time.go`, `pull_utils.go` (`appendPullChangesFromRows`, `pullChangeFromSnapshot` — в `syncdomain`). В корне остаются `handler.go`, `push.go`, `pull.go` и общие интеграционные тесты на transport.

Внутри шагов новые тесты не пишутся — соответствие ADR-0003. Существующие unit-тесты переезжают вместе с production-кодом.

## Последствия

- **Навигация по сущности линейна.** Логика `plan` целиком в `sync/plan/plan.go`. Логика общая — в `syncdomain/`. Логика transport — в корне.
- **Корневой `sync/` сокращается с 26 до 3 файлов** (+ интеграционные тесты).
- **Backfill ADR-0003 ведётся в подпакетах.** Новые тесты не раздувают корень.
- **Параллельная работа над разными сущностями не конфликтует** в push.go/pull.go: каждая сущность редактируется в своём подпакете, transport-файлы стабильны.
- **Подпакеты получают публичный API.** `project.ApplyPush`, `syncdomain.PushStepResult` — это типы пакета, не unexported helpers. Граница чётче, но публичная поверхность кода растёт.
- **`PushStepResult` и `InternalPushErr` становятся exported.** Это внутренний контракт между подпакетом сущности и transport, не контракт клиента. Контракт клиента (`PushConflict`/`PushError`) живёт в `synctypes` и не меняется.
- **Цена перехода — 7 шагов.** В сумме это сопоставимо с шагами 1–2 ADR-0004 (которые отменяются) плюс ~5 шагов миграции сущностей. Каждый шаг минимален и обратим.
- **ADR-0004 теряет применимость для `sync/`.** Правила 1–5 ADR-0004 действуют для `auth/`, `photos/` и любых новых handler-пакетов, не превышающих порог количества файлов. Для пакетов выше порога применяется ADR-0005.

## Открытые вопросы

1. **Порог «количество файлов в пакете → разбиение по сущности».** ADR-0005 применён к `sync/` post factum по очевидности (26 → 34 файла). Формальный порог (например, «≥15 файлов» или «≥3 сущности с собственными push/pull») в этом ADR не закрепляется. Триггер пересмотра: появление третьего handler-пакета, для которого встаёт вопрос разбиения, — тогда порог формализуется отдельным ADR на основе двух прецедентов.
2. **Узкие per-domain интерфейсы вместо общего `db.Querier`.** Унаследовано как открытый вопрос из ADR-0004 (пункт 3). Подпакет `sync/plan/` принимает общий `db.Querier`, а не `PlanRepo`. Триггер пересмотра: fake `db.Querier` становится узким местом сопровождения или появляется второй адаптер БД.
3. **Перенос `synctypes` в `syncdomain` или наоборот.** `synctypes` — контракт клиента (`PullChange`, `PushOperation`, `PushConflict`, `PushError`). `syncdomain` — внутренние правила. Сейчас они в разных пакетах, и подпакеты сущности импортируют оба. Объединение упростит граф импортов, но смешает «контракт клиента» и «внутренний контракт». Триггер пересмотра: появление второго sync-протокола (например, для админских операций), где `synctypes` и `syncdomain` начнут расходиться сильнее.
