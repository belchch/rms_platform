# ADR 0004: Расслоение handler-пакетов

## Статус

Принято — 2026-05-14. Применимость к пакету `sync/` снята ADR-0005 (2026-05-14): для `sync/` действует разбиение по сущности, не файловая раскладка по слоям. Правила 1–5 этого ADR остаются в силе для `auth/`, `photos/` и любых новых handler-пакетов ниже порога разбиения.

## Контекст

Каждый домен в `internal/handler/<domain>/` исторически складывался индивидуально. Как результат — три существующих пакета имеют три разных шейпа:

- `sync/` — 15 production-файлов, 5 сущностей × 3 операции. Orchestration-функции висят на ресивере `*handler`, который им не нужен; чистая логика (LWW, валидация workspace) лежит в `push.go` рядом с транзакционным кодом.
- `auth/` — один файл на 240 строк. Часть orchestration уже вынесена в package-level (`rotateRefreshToken`), часть — в методы на `*handler` (оправданно, нужны `q`, `pool`, `jwtSecret`). Чистые helpers (`hashRefreshToken`, `random*`) рядом, но не отделены файлово.
- `photos/` — один файл на 92 строки, один endpoint, валидация inline.

Без общего правила каждый новый handler-пакет будет заведён в произвольной форме. Неоднородность мешает чтению и провоцирует «куда положить функцию» при каждой правке.

Проблема не в конкретном `sync/`, а в **отсутствии конвенции** на расположение слоёв внутри handler-пакета.

Цель конвенции — минимальный I/O-код без бизнес-логики, изолированная бизнес-логика и предсказуемое место для unit-тестов. Handler-пакет делится на три зоны ответственности: тонкий transport-слой, orchestration поверх `db.Querier` и pure domain без I/O. I/O-слой проверяется интеграционно, а логика тестируется через pure-функции или fake `db.Querier`.

## Решение

Конвенция применяется ко всем пакетам `internal/handler/<domain>/`. Состоит из пяти правил.

### Правило 1. Три слоя внутри пакета

| Слой | Что включает | Зависимости |
|---|---|---|
| Transport | huma-handler'ы, `Register`, типы `*Input`/`*Output`, `handler{}`, начало транзакции | `huma.API`, `*pgxpool.Pool`, middleware |
| Orchestration | дисптач операций, координация чтений и мутаций, кросс-сущностные helpers | `db.Querier`, transport не использует |
| Pure domain | инварианты, валидаторы, snapshot-маперы, чистые расчёты (LWW и т. д.), системные примитивы (`crypto/rand`, генерация UUID/opaque token, `time.Now()`) | без I/O состояния приложения, без Querier, без huma |

Слои — концептуальные. Они отражаются в имени файлов (Правило 3), а не в подпакетах.

Системные примитивы в pure domain допустимы, если функция не читает и не меняет состояние приложения, не требует fake-инфраструктуры и проверяется unit-тестом по свойствам результата.

### Правило 2. Ресивер `*handler` — только на transport, и только если используется

Ресивер `*handler` допустим **исключительно** на huma-handler'ах (`push`, `pull`, `signIn`, `refresh`, `uploadUrl` и подобных), и **только** если функция читает хотя бы одно поле `h.*`. Если поле не используется — функция переводится в package-level и принимает зависимости параметрами.

На orchestration- и domain-функциях `*handler` ресивер запрещён. Зависимости (`db.Querier`, секреты, конфиг) передаются параметрами.

Структура handler внутри пакета называется `handler` с lowercase. Пакет наружу предоставляет `Register(api, ...)`, а не concrete handler-тип.

Конкретные следствия:

- `sync.applyPushOperation`, `sync.pushProject*`, `sync.pushPlan*`, `sync.pushRoom*`, `sync.pushWall*`, `sync.pushPhoto*` — должны стать package-level (поле `h` не читают).
- `auth.signIn`, `auth.refresh` — остаются методами (читают `h.q`, `h.pool`, `h.jwtSecret`).
- `photos.uploadUrl` — остаётся методом (читает `h.presigner`).

### Правило 3. Файлы именуются по слою

Базовая раскладка в любом handler-пакете:

| Файл | Слой | Содержимое |
|---|---|---|
| `handler.go` | transport | `handler{}`, `Register`, `bearerAuth`, типы `*Input`/`*Output` |
| `<endpoint>.go` (`push.go`, `pull.go`, `signin.go`, …) | transport | huma-handler одного endpoint'а |
| `service.go` | orchestration | кросс-сущностные orchestration-функции, `pushStepResult`, `internalPushErr`-подобные helpers |
| `domain.go` | pure | чистые правила пакета |
| `<entity>.go` (`push_project.go`, `pull_project.go`, …) | orchestration + per-entity pure | orchestration сущности и её snapshot-функция |

`<entity>.go` — компромисс ради cohesion: вся логика одной сущности читается одним файлом, даже если внутри есть pure-функция (snapshot). Это допустимо, потому что pure-функция привязана к конкретной сущности и за её пределами не используется.

### Правило 4. Порог применимости

Раскладка на отдельные `service.go` и `domain.go` обязательна, когда в пакете одновременно выполнены **оба условия**:

1. ≥1 нетривиальная orchestration-функция (запрашивает `Querier`, координирует ≥2 шага).
2. ≥1 чистая функция, не привязанная к одной сущности (например, общий валидатор, общий маппер).

Если оба условия не выполнены, допустимо держать пакет в одном файле `handler.go`. Это нужно, чтобы крошечные пакеты вроде `photos/` (92 строки, 1 endpoint) не дробились формально ради конвенции.

При нарушении порога после роста пакета — раскладка делается отдельным PR, **до** добавления новой функциональности в этот пакет.

### Правило 5. Обработка ошибок по слоям

Pure domain возвращает типизированные результаты или sentinel-ошибки. Pure domain не возвращает `huma.NewError`, не выбирает HTTP-код и не пишет в logger.

Orchestration транслирует доменные сигналы и sentinel-ошибки после чтения из БД (например, `errors.Is(err, pgx.ErrNoRows)`) в контрактные значения пакета: `synctypes.PushError`/`synctypes.PushConflict` для `sync/`, `huma.StatusError` для `auth/` и аналогичные transport-контракты для других handler-пакетов. Технические сбои, не подлежащие контракту клиента, оборачиваются через `fmt.Errorf("operation: %w", err)`.

Transport не интерпретирует ошибки orchestration. Huma-handler возвращает результат orchestration как есть, а HTTP-представление строит huma по `StatusError`.

Логирование технических ошибок находится на orchestration-границе, рядом с местом наблюдения. Pure domain и transport не дублируют logging.

## Применимость к существующим пакетам

| Пакет | Соответствие сейчас | Что нужно сделать |
|---|---|---|
| `sync/` | вне области ADR-0004 (см. Статус). Применимость передана ADR-0005: разбиение по сущности вместо файловой раскладки. | См. раздел «Применение» ADR-0005. |
| `auth/` | соответствует Правилам 1, 2, 4. Нарушено Правило 3 (один файл `handler.go` на 240 строк, превышает порог). Правило 5 частично соблюдается: `rotateRefreshToken` возвращает `huma.StatusError`, а helpers чистые. | Создать `service.go` (`rotateRefreshToken`) и `domain.go` (`hashRefreshToken`, `random*`). Ресиверы на `signIn`/`refresh` сохраняются. |
| `photos/` | соответствует Правилу 4 (ниже порога) и не требует раскладки. Частично нарушает Правило 2: concrete handler-тип называется `Handler`, хотя наружу пакет предоставляет только `Register`. | Переименовать `Handler` в `handler` отдельным маленьким PR. Пересмотр раскладки — при превышении порога. |

## Применение

Применение к `sync/` снято ADR-0005. Шаги для `auth/` и `photos/` выполняются по этому ADR независимо.

1. **`auth/` — раскладка по файлам.** Создаются `service.go` (`rotateRefreshToken`) и `domain.go` (`hashRefreshToken`, `random*`). `handler.go` оставляет `handler{}`, `Register`, методы `signIn`/`refresh`, типы `*Input`/`*Output`, константы.
2. **`photos/` — имя concrete handler-типа.** `Handler` переименовывается в `handler`. Без файловой раскладки: пакет ниже порога Правила 4.

Каждый шаг самостоятелен, проверяется через `just check` и мёрджится отдельно. Внутри шагов новые тесты не пишутся — соответствие ADR-0003 о разделении архитектурного сдвига и написания тестов.

## Последствия

- **Единый шейп пакета.** При чтении любого `handler/<domain>/` слой определяется по имени файла. При добавлении нового handler-пакета конвенция говорит, как его раскладывать.
- **Pure-уровень тестируется без фейка.** `domain.go` импортируем без зависимости на `Querier`/huma, чистые функции получают unit-тест без моков. Согласовано с `pure-vs-io.mdc`.
- **Orchestration тестируется через `Querier`-фейк** — без изменений (ADR-0002).
- **Нет нового публичного API.** Внешние импорты пакетов идут через `Register(api, ...)`. `cmd/api/main.go` не меняется.
- **Не закрывает** вынос транзакционного управления (см. «Открытые вопросы» ADR-0002). Ресивер `*handler` транспорта по-прежнему держит `*pgxpool.Pool`, тестируется только интеграционно.
- **Порог применимости — явная мера** против формального дробления крошечных пакетов.
- **Общий `db.Querier` остаётся крупным seam.** Fake для orchestration-тестов может расти быстрее production-кода. Узкие per-domain интерфейсы сознательно не вводятся этим ADR.

## Открытые вопросы

1. **Перенос в подпакеты для `sync/`.** Закрыт ADR-0005 (разбиение по сущности). Для `auth/` и `photos/` остаётся открытым: триггер пересмотра — `service.go` или `domain.go` в одном из пакетов становится трудно читаемым, либо появляется второй потребитель чистых функций. Решение оформляется отдельным ADR.
2. **Mat Ryer-стиль (`run(ctx, args, getenv, ...)` + `NewServer`)** — не входит в этот ADR. Если будет принят отдельным ADR ради e2e-тестов через `run()`, текущая конвенция с ним не конфликтует: `Register(api, ...)` остаётся точкой входа в `NewServer`.
3. **Узкие per-domain интерфейсы.** ADR-0004 продолжает использовать общий `db.Querier` из ADR-0002. Введение узких интерфейсов (`RefreshTokenRepo`, `PhotoRepo`, `PlanRepo` и т. д.) даёт ISP и сокращает fake'и, но удваивает количество типов и не даёт прироста уверенности в текущем backfill. Вопрос пересматривается, когда fake `db.Querier` становится узким местом сопровождения или появляется второй адаптер БД.
4. **Новые handler-пакеты** (если появятся) — заводятся сразу по конвенции. Пересмотр конвенции — отдельным ADR.
