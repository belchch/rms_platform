# ADR 0003: Переходный режим — обкладывание существующей базы тестами

## Статус

Принято — 2026-05-12. Активен.

## Контекст

Базовый код проекта генерировался без стадии проектирования (контракты, декомпозиция, adversarial-проход). Тесты появились позже, задним числом. На момент принятия этого ADR:

- Statement coverage: 10.8% (после исключения `internal/db/`) — см. `docs/planning/testing-baseline.md`.
- Endpoint coverage: 1 из 6 маршрутов.
- Критические инварианты `auth` и `sync` не покрыты ни одним тестом.

Попытки написать тесты уже дважды обнаружили архитектурные проблемы и потребовали структурных изменений:

1. **ADR-0002 (Querier seam)** — зависимость на `*db.Queries` делала handler нетестируемым; решение — переход на `db.Querier`.
2. **Вынос `time.go`** (коммит `6aec11c`) — сравнение `now()` vs `clientUpdatedAt` было вмешано в handler; решение — extraction чистой функции по `pure-vs-io.mdc`.

Это формирует паттерн: тест как зонд, обнаруживающий нарушение границ, а не просто проверяющий поведение.

Цель переходного режима — пройти от текущего состояния к `ai-dev-workflow` (проектирование + контракты + тесты per-фича), не накапливая дальнейший долг.

## Решение

### Приоритизация

На время режима тесты существующего кода приоритетнее новых фичей. Новые задачи принимаются только:

- как согласованные багфиксы,
- как задачи, напрямую разблокирующие написание теста.
- или в случае крайней необходимости 

### Выбор кандидата для покрытия

Кандидат принимается, если:

1. Инвариант формулируется в одном предложении.
2. Кандидат лежит в горячем пути: `internal/handler/auth`, `internal/handler/sync`, `internal/handler/photos`, `internal/middleware`, `internal/jwtutil`.
3. Есть либо чистая функция (unit-тест без моков), либо явный seam (`db.Querier`, `storage.Provider`).

Кандидат отклоняется (стоп-лист):

- `internal/db/*` — сгенерированный код sqlc.
- Тонкие мапперы, чистые DTO, structs без логики.
- Тривиальные обёртки с единственной строкой делегирования.

Выбор по инварианту и цене бага, не по проценту покрытия — см. `.cursor/rules/testing-goals.mdc`.

### Разрешение на архитектурный сдвиг

Если ради тестируемости требуется seam или extraction чистой функции — это **разрешено** при условии:

- Сдвиг оформляется отдельным ADR (прецеденты: ADR-0002, `time.go`).
- Сдвиг без ADR является hack'ом по `.cursor/rules/no-hacks.mdc`.
- Архитектурный сдвиг не смешивается с написанием самого теста в одном PR.

## Definition of Done переходного режима

Режим считается завершённым, когда одновременно выполнены:

`**auth`**

- Ротация refresh-токена: после `/auth/refresh` старый токен инвалидирован.
- Хеширование: в `refresh_tokens` хранится хеш, а не plaintext.
- Валидность access claims: истёкший или поддельный токен → 401.
- Изоляция workspace: запрос с токеном workspace A не видит данные workspace B.

`**sync**`

- LWW: операция с `clientUpdatedAt` <= `db.updated_at` попадает в `conflicts[]`, не в `applied[]`.
- Монотонность `sync_cursor`: каждый push увеличивает cursor; pull с `since=N` не возвращает строки с cursor <= N.
- Soft-delete как tombstone: удалённая строка возвращается в pull с `deleted_at != null`.
- Изоляция workspace: pull/push не смешивают данные разных workspace.

**Интеграционные тесты для `pull*`**

Код `internal/handler/sync/pull*.go`, вызывающий `List*Since`, критичен для инвариантов выше unit-уровня Go и должен явно проверяться против Postgres: фильтр `workspace_id`, пагинация `sync_cursor > since`, возврат soft-deleted строк без `deleted_at IS NULL` в запросе, согласованность payload. Покрытие только через unit и fake `Querier` не заменяет этот этап; его планируют после текущего backfill unit-тестами для `push*`.

**Метрики**

- Endpoint coverage >= 50% (3 из 6 маршрутов из `testing-baseline.md`).

## Последствия

- Раздел «Transitional mode» в `.cursor/rules/testing-goals.mdc` живёт ровно до закрытия этого ADR.
- При закрытии: обновить статус этого файла на «Завершено», удалить раздел из `testing-goals.mdc`, обновить `docs/adr/README.md` — одним коммитом.
- Linear Project «Test backfill — переходный режим» закрывается синхронно.
- После выхода из переходного режима новые фичи разрабатываются по `ai-dev-workflow`: проектирование → контракты → код → тесты.
