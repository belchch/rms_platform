# Архитектурные правила

## 2026-05-10

- Handler-функции принимают `db.Querier` (интерфейс sqlc), не `*db.Queries` (конкретный struct). `*db.Queries` собирается только в `cmd/api/` и в `db.New(tx)` при старте транзакции внутри handler'а.
- В `apps/api/sqlc.yaml` обязателен `emit_interface: true` — это создаёт `Querier` как явный шов между handler-слоем и persistence-слоем.
- Чистая логика (валидация payload, сравнение значений, выбор стратегии) выносится из handler-функций в отдельные функции, не зависящие от БД. Симптом нарушения: для тестирования чисто логической ветки требуется живой Postgres.
- Тесты пишутся под инвариант, не под coverage. Перед написанием теста — вопрос: что сломается без него, есть ли нетривиальная логика.
- LWW-сравнение таймстампов строгое (`clientMs > serverMs`). При равенстве сервер не уступает.
- Функции, читающие `pgtype.Timestamptz` из БД, явно проверяют `Valid` и возвращают ошибку при `Valid = false`. Молчаливый fallback на эпоху Unix запрещён.
- Архитектурное решение, меняющее правила взаимодействия слоёв (замена конкретного типа на интерфейс, изменение управления транзакциями), оформляется ADR-ом в `docs/adr/`, а не точечным коммитом.
