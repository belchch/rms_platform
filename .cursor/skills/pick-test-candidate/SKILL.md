# Выбор кандидата для теста

Применяется во время переходного режима (ADR-0003).
Горячие пакеты: `internal/handler/auth`, `internal/handler/sync`,
`internal/handler/photos`, `internal/middleware`, `internal/jwtutil`.

## Шаг 1 — получить актуальное покрытие

```
just check-coverage
```

Из вывода `go tool cover -func` взять покрытие **каждой функции** в горячих пакетах.

## Шаг 2 — отфильтровать стоп-лист

Исключить до оценки:
- `internal/db/*` — сгенерированный sqlc
- Одностроковые делегаты и конструкторы-обёртки
- Чистые DTO и мапперы без ветвления
- Тривиальные чистые функции с очевидным контрактом, где тест не добавит уверенности сверх прочтения кода за секунды — нарушает принцип «тестируйте поведение, а не методы». Симптомы: одна `if`-ветка, фиксированная строка ошибки, отсутствие нетривиальной логики
- `cmd/api/main.go`, `config.Load` — wiring без логики

## Шаг 3 — эмпирический скоринг каждого кандидата

**Критерии выбора** (не тестируемость):

| Поле | Шкала | Как оценивать |
|---|---|---|
| **Цена ошибки** | 1–5 | 5 = auth bypass / data leak чужого workspace; 3 = неверный результат; 1 = cosmetic |
| **Вероятность ошибки** | 1–5 | 5 = ветка не покрыта и код менялся недавно; 1 = стабильная функция, ни разу не трогали |
| **Приоритет** | цена × вероятность | итоговый score, максимум 25 |

**Поля тестируемости** (влияют на способ, не на приоритет):

| Поле | Возможные значения |
|---|---|
| **Тип теста** | `unit` / `integration` / `unit после архитектурного сдвига` |
| **Сдвиг нужен?** | да / нет — если логика и I/O слиплись (`pure-vs-io.mdc`) или нет seam |
| **Прирост покрытия** | оценочный % statements, которые закроет тест |

## Шаг 4 — построить таблицу кандидатов

Колонки: `Функция | Покрытие сейчас | Инвариант | Цена | Вероятность | Score | Тип теста | Сдвиг? | Прирост %`.

Заполнить строку для каждого кандидата, прошедшего стоп-лист. Отсортировать по `Score` убыванием.

## Шаг 5 — вывод

Для топ-3 объяснить:
1. Почему высокий score именно у этого кандидата.
2. Что не так с вариантами ниже в списке (низкий score или стоп-лист).
3. Если нужен архитектурный сдвиг — сказать явно: "сначала ADR, потом тест, два отдельных PR".
