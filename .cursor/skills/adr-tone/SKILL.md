# ADR style

ADR фиксирует **целевое состояние** или **правило**, не задачу на рефакторинг. Читатель ADR должен узнать, что становится истинным, а не что кто-то делает.

## Правило области

ADR описывает класс случаев, не одиночный кейс. Если правило применимо ровно к одному пакету/файлу/коммиту — это не ADR. Допустимые исходы:

1. Перепроверить scope: возможно, правило обобщается до класса (все handler-пакеты, все sqlc-операции, все sync-эндпоинты).
2. Расширить ADR до класса; одиночный кейс становится первым применением.
3. Отказаться от ADR в пользу комментария в коде, описания PR или раздела в `.cursor/rules/*.mdc`.

## Обязательная структура

Каждый ADR содержит разделы в этом порядке:

| Раздел | Содержит | Не содержит |
|---|---|---|
| Заголовок | формулировку правила, конвенции или инварианта | глагольные или отглагольные имена действия |
| Статус | принято/отвергнуто/заменено + дата ISO | "в работе", "обсуждается" |
| Контекст | симптомы, фактическое состояние, что провоцирует решение | "хотим", "планируем", "решили попробовать" |
| Решение | набор правил/инвариантов в настоящем времени | план шагов, последовательность действий, перечень коммитов |
| Применимость | классификация всех существующих случаев класса (соответствует / нарушено / нарушено частично + что делать) | пропуск раздела, если в коде есть случаи класса |
| Применение | план перехода от текущего состояния к целевому | смешение с разделом «Решение» |
| Последствия | что меняется по факту — положительное и отрицательное | реклама, односторонние выгоды |
| Открытые вопросы | сознательно отложенное с триггером пересмотра | "не знаем", "подумаем" |

Раздел «Применимость» обязателен, если в кодовой базе уже существуют случаи класса, к которому относится правило. Раздел «Применение» обязателен, если решение требует изменений в существующем коде.

## Правила тона

1. Решение пишется в настоящем времени, страдательном или нормативном залоге. Запрещены глаголы первого лица («мы …»), будущее время, сослагательное наклонение.
2. Глаголы действия допустимы только в разделе «Применение».
3. Запрещены формулировки: «хотим», «планируем», «возможно», «в будущем», «стоит подумать», «постараемся», «решили попробовать».
4. В «Открытых вопросах» обязательна формулировка триггера пересмотра: при каком факте/событии вопрос становится решаемым.
5. Cross-refs на `.cursor/rules/*.mdc` и предыдущие ADR обязательны там, где решение опирается на них или их продолжает.
6. Заголовок и имя файла — формулировка правила, не действия.

## Правило независимости пунктов решения

Каждый пункт раздела «Решение» проверяется независимо: можно сказать «соответствует» или «нарушено» для одного пункта без чтения остальных. Если два пункта читаются только вместе — объединить или переформулировать.

## Правило симметрии последствий

В разделе «Последствия» называется минимум одно отрицательное последствие или явно сказано, что отрицательных нет и почему. ADR без отрицательных последствий допустим только при чисто формализующем решении (фиксация существующей практики без изменений в коде).

## Правило: ADR не нумерует PR

Раздел «Применение» описывает **порядок переходных шагов** и инварианты между ними, не план задач трекера.

Запрещено:

- нумеровать пункты как «PR-1», «PR-2», «PR-N»;
- использовать слово «PR» как идентификатор шага;
- ссылаться на номера PR/коммитов/issue внутри ADR.

Разрешено:

- единичная вводная фраза «реализуется N последовательными шагами/PR», без последующей нумерации этих PR;
- нумерованный список шагов с формулировками в настоящем времени или нормативном залоге («создаются `service.go` и `domain.go`», «ресиверы снимаются»);
- описание зависимостей между шагами («шаг 2 возможен только после шага 1, потому что …»).

Обоснование: ADR живёт пока действует правило, PR — до мёрджа. Список конкретных PR превращает ADR в чек-лист задачи и устаревает сразу после реализации. Обратная связь идёт через PR-описания и issue tracker: PR ссылается на ADR, не наоборот.

## Регистрация и нумерация

1. Номер — следующий за последним по `docs/adr/`.
2. Файл: `docs/adr/NNNN-kebab-case-name.md`. Имя — формулировка правила.
3. После создания — добавить строку в `docs/adr/README.md`. Текст ссылки совпадает с заголовком ADR.
4. Если ADR заменяет предыдущий — старый получает статус «Заменено ADR-NNNN», в новом указано «Заменяет ADR-MMMM».

## Self-audit перед финализацией

Перед записью файла пройти по списку. Каждый пункт — да/нет. Любое «нет» требует исправления.

1. Заголовок — формулировка правила, не действия.
2. Контекст описывает проблему, не намерение.
3. Решение в настоящем времени, без процедурных глаголов.
4. Каждый пункт решения применим ко всем существующим случаям класса.
5. Раздел «Применимость» покрывает все существующие случаи класса в кодовой базе (если есть).
6. Раздел «Применение» отделён от «Решения» отдельным заголовком.
7. Раздел «Применение» не нумерует PR и не ссылается на номера PR/коммитов/issue.
8. Раздел «Последствия» содержит хотя бы одно отрицательное последствие или явное «нет, потому что …».
9. Каждый «Открытый вопрос» содержит триггер пересмотра.
10. Cross-refs на `.cursor/rules/*.mdc` и связанные ADR расставлены.
11. Запрещённые формулировки (см. «Правила тона», п. 3) отсутствуют.
12. `docs/adr/README.md` обновлён.

## Связанные материалы

- `.cursor/rules/no-hacks.mdc` — требование root-cause; ADR — основной артефакт фиксации root-cause решений.
- `docs/adr/0003-test-backfill-transitional.md` — пункт «архитектурный сдвиг оформляется отдельным ADR» как формальное основание для создания ADR.
- `~/.cursor/skills/skills-no-examples/SKILL.md` — соседний override, действующий на SKILL.md, не на ADR; ADR может содержать конкретику.
